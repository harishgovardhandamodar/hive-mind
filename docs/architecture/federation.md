# Federation

The `Federation` class manages all hives and provides cross-graph operations.

## Meta-graph

A lightweight `MultiDiGraph` where:

- **Nodes** are hives (referenced by graph ID)
- **Edges** are links between hives

The meta-graph is stored in `data/meta_graph.json` and persisted as JSON.

## Registration

Hives are discovered automatically by scanning the `hives/` directory for `knowledge_graph.json` files. They can also be registered programmatically via `register_graph()`.

## Cross-graph edges

When two concepts in different hives are connected, the edge is stored in the source hive's graph with:

```json
{
  "source": "concept:GNN",
  "target": "concept:GraphNetwork",
  "relation": "related_to",
  "cross_graph": true,
  "target_graph": "graph-networks"
}
```

These appear in the dashboard as dashed yellow lines.

## Multi-hop queries

When a user types a natural language query mentioning hive names (e.g., "How is GNNs related to LLMs?"), the federation:

1. Identifies hive names using case-insensitive matching against registered graph IDs
2. If exactly 2 hives are found, runs BFS on the meta-graph to find the shortest path
3. If 3+ hives are found, uses them as waypoints
4. Collects all nodes from the source and target hives
5. Collects cross-edges and concept nodes from intermediate hives
6. Returns the combined subgraph with a human-readable explanation

## Unified search

`unified_search()` scores all nodes across all hives using TF-IDF with an exact-match bonus. Results include the `graph_id` so you can see which hive each result belongs to.

## Peer-to-peer sharing

HiveMind instances can share hives with each other over the internet through a peer-to-peer pairing system.

### Instance identity

Each instance generates a persistent `instance_id` (UUID) and `instance_secret` (256-bit HMAC signing key) on first launch, stored in `data/peers.json`. The `instance_info()` method exposes these (excluding the secret) for the pairing handshake.

### Peer registry

Peers are stored in `data/peers.json` alongside the instance identity. Each peer entry includes:
- `id` — short local identifier
- `name` — friendly name or URL
- `url` — the peer's public URL
- `added_at` — ISO timestamp
- `fingerprint` — peer's identity fingerprint (optional)

### Pairing flow

Pairing is **bidirectional** — both instances register each other:

1. **Initiator** fetches `/api/peering/info` from the target instance
2. Validates not self (compares `instance_id`)
3. Checks `find_peer_by_url()` to avoid duplicates
4. Registers the target as a local peer via `add_peer()`
5. Calls the target's `/api/peering/pair` with its own info
6. Target registers the initiator via `accept_pair()`

### Secure pairing with invite tokens

When TLS is not available, pairing requires a **one-time invite token** generated by the target instance:

1. Target runs `hivemind peers invite` → gets HMAC-SHA256 signed token
2. Token format: `uuid:expiry_timestamp:hmac_signature` (first 12 hex chars of HMAC)
3. Token is shared out-of-band with the initiator
4. Initiator includes the token in the pairing request
5. Target verifies the token — checks HMAC, expiry, and one-time use
6. Used tokens are tracked in memory to prevent replay

`create_invite(ttl=600)` generates tokens. `verify_invite()` validates them using `hmac.compare_digest()`.

### TLS support

HTTPS can be enabled with `--cert` and `--key` flags on `serve()`. When both instances use HTTPS, pairing does not require a token since TLS provides transport security. The server uses `ssl.SSLContext` with `PROTOCOL_TLS_SERVER` to wrap the socket.

### Export/import

- `export_hive_data(hive_id)` serializes a hive as `nx.node_link_data` JSON with `hivemind_export` metadata
- `import_hive_data(data, target_name, merge_similar=True)` reconstructs a `KnowledgeGraph`, optionally merging similar concepts (Jaccard ≥ 0.5) with cross-graph edges
- `pull_peer_hives(peer_id, hive_id=None)` fetches and imports hives from a peer
